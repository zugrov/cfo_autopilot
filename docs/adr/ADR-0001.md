# ADR-0001: Deterministic Forecast Formula (не ML)

**Статус:** Принято  
**Дата:** 2026-05-23  
**Авторы:** Engineering Review

---

## Контекст

Нужно предсказывать остаток денег на горизонте 7–91 день для МСБ в РФ.
Данные: банковские транзакции + ручные обязательства + (с Sprint 2) дебиторка.

Варианты рассматривались:
1. **Детерминированная формула** — арифметика по известным обязательствам и медиане входящих.
2. **ML/ARIMA** — авторегрессия по историческим данным.

---

## Решение: Детерминированная формула

**Sprint 1 (bank-only):**
```
forecast_balance[d] = balance[d-1]
                    + median_daily_inflow_7d      # fallback: нет истории → 0
                    - planned_obligations[d]       # ручной ввод
```

**Sprint 2+ (полная формула):**
```
forecast_balance[d] = balance[d-1]
                    + planned_inflows[d]
                    + expected_receivable_collections[d]
                    - planned_obligations[d]
                    - recurring_pattern[d]
```

Сценарии (коэффициенты): base=1.0, stress: inflow×0.85 / outflow×1.1, optimistic: inflow×1.1 / outflow×0.9.

---

## Обоснование

- **Auditable:** собственник видит, из чего складывается цифра. «Медиана за 7 дней ₽120k + обязательство ₽80k = остаток ₽40k».
- **Нет data cold-start:** ML требует ≥6 месяцев данных; у нового клиента их нет.
- **Предсказуемость ошибки:** формула ошибается явно (забытые обязательства), ML — непрозрачно.
- **Solo dev:** ML-пайплайн (обучение, версионирование моделей, drift detection) избыточен для MVP.

---

## Последствия

- Точность ограничена качеством введённых обязательств → UI badge «прогноз неполный».
- При отсутствии `planned_obligations` → flat balance → ложное спокойствие → **обязательный** CTA «Добавьте обязательства».
- В будущем можно добавить LLM-assisted categorization для автозаполнения recurring patterns.
