# ADR-0005: ARQ (Redis) для асинхронных задач

**Статус:** Принято  
**Дата:** 2026-05-23  
**Авторы:** Engineering Review

---

## Контекст

Продукту нужны фоновые задачи:
- Пересчёт `daily_cash_snapshot` после загрузки файла.
- Ежедневный Telegram-дайджест в 08:00 MSK.
- (Sprint 2) Alert engine — проверка дефицита по расписанию.
- (Sprint 3) LLM-запросы с timeout handling.

Варианты:
1. **FastAPI BackgroundTasks** — встроено, просто.
2. **ARQ (Redis)** — легковесная очередь задач на asyncio/Redis.
3. **Celery** — battle-tested, но тяжёлый для solo dev.
4. **Temporal/Prefect** — избыточно для MVP.

---

## Решение: ARQ (Redis)

```python
# app/workers/settings.py
from arq import cron
from arq.connections import RedisSettings

class WorkerSettings:
    functions = [recompute_snapshot, send_telegram_digest]
    cron_jobs = [
        cron(send_telegram_digest, hour=5, minute=0)  # 08:00 MSK = 05:00 UTC
    ]
    redis_settings = RedisSettings.from_dsn(settings.REDIS_URL)
```

---

## Обоснование выбора ARQ

| Критерий | BackgroundTasks | ARQ | Celery |
|---|---|---|---|
| Выживает при рестарте процесса | Нет | Да (Redis) | Да |
| Зависимости | 0 | Redis | Redis + Celery |
| asyncio native | Да | Да | Частично |
| Retry / dead letter | Нет | Да | Да |
| Подходит для solo dev MVP | Да | Да | Избыточно |

**BackgroundTasks отклонён:** задача теряется при рестарте контейнера (crash, deploy).
Для финансового импорта потеря задачи = потеря состояния пользователя → критично.

**Celery отклонён:** отдельный worker процесс, beat scheduler, flower monitoring — 3× overhead.

---

## Последствия

- Redis обязателен с Sprint 1b (можно использовать Upstash Redis или Redis в YC).
- `docker-compose.yml` включает `redis:7-alpine`.
- ARQ worker запускается отдельным процессом: `arq app.workers.settings.WorkerSettings`.
- Задачи: `recompute_snapshot(company_id)`, `send_telegram_digest(company_id)`.
- Retry: 3 попытки с экспоненциальным backoff (ARQ default).
