# ADR-0003: PostgreSQL Row-Level Security для мультитенантности

**Статус:** Принято  
**Дата:** 2026-05-23  
**Авторы:** Engineering Review

---

## Контекст

SaaS с несколькими компаниями (tenant = company). Один инстанс PostgreSQL, одна БД.
Критичное требование: утечка данных между компаниями — критический инцидент (152-ФЗ, финансовые данные).

Варианты изоляции:
1. **PostgreSQL RLS** — политики на уровне БД, `company_id = current_setting('app.company_id')`.
2. **Application-level filter** — `WHERE company_id = ?` в каждом запросе.
3. **Schema-per-tenant** — отдельная схема на компанию.

---

## Решение: PostgreSQL RLS

На каждой tenant-таблице:

```sql
ALTER TABLE transactions ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON transactions
    USING (company_id = current_setting('app.company_id')::uuid);
```

Настройка сессии в FastAPI middleware:

```python
async def set_tenant(conn, company_id: str):
    await conn.execute(
        "SELECT set_config('app.company_id', $1, true)", company_id
    )
```

Таблицы с RLS: `transaction`, `import_batch`, `bank_account`, `obligation`,
`daily_cash_snapshot`, `audit_log`, `counterparty`, `receivable`, `alert`.

---

## Обоснование

- **Defense in depth:** даже при баге в application-layer, БД не вернёт чужие данные.
- **Supabase Auth интеграция:** Supabase поддерживает RLS natively, JWT claim `company_id` доступен через `auth.uid()` и custom claims.
- **Аудит:** проще доказать изоляцию на уровне БД (PCI DSS, 152-ФЗ).
- **Schema-per-tenant** отклонён: миграции ×N компаний, сложность мониторинга.
- **App-level filter** отклонён: единственная линия защиты, уязвима к ошибкам разработчика.

---

## Последствия

- Каждый запрос требует `set_config('app.company_id', ...)` до первого SELECT — middleware обязателен.
- Тест: `user A cannot read company B (403)` — в integration test suite.
- `audit_log` пишется с Sprint 1a; политика `INSERT` — без ограничений (пишет сервис, не пользователь).
- Суперпользователь (admin операции) — отдельная роль с `BYPASSRLS`.
